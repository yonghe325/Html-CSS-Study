每次 MutationRecord 被添加到 MutationObserver 的记录队列时
仅当之前没有已排期的微任务回调时（队列中微任务长度为 0）
才会将观察者注册的回调（在初始化 MutationObserver 时传入）
作为微任务调度到任务队列上
这样可以保证记录队列的内容不会被回调处理两次
不过在回调的微任务异步执行期间，有可能又会发生更多变化事件
因此被调用的回调会接收到一个 MutationRecord 实例的数组，顺序为它们进入记录队列的顺序
回调要负责处理这个数组的每一个实例，因为函数退出之后这些实现就不存在了
回调执行后，这些 MutationRecord 就用不着了，因此记录队列会被清空，其内容会被丢弃
# 记录队列
```js dp
// 想象有一个“待办清单”（记录队列）
const todoList = [];

// 每次 DOM 变化，就在清单上记一笔
todoList.push('属性改变了');
todoList.push('子节点添加了');
todoList.push('文本修改了');
```
# 微任务调度
```js dp
// 关键逻辑：
// 只有当“待办清单”从空变为非空时，才派一个“打包员”（微任务）去处理

if (todoList.length === 0) {
  // 刚刚有了第一个待办事项
  scheduleWorker();  // 派一个打包员（调度微任务）
}
todoList.push(newChange);  // 继续添加事项
```
# 整个过程
```js dp
时间线：
─────────────────────────────────────────────────────►

DOM变化1: 修改属性
    ↓
[记录队列] 从 [] 变为 [变化1]  ← 第一次有内容！
    ↓
📌 调度一个微任务（派一个打包员）  ← 只调度这一次
    ↓
DOM变化2: 添加子节点
    ↓
[记录队列] [变化1, 变化2]  ← 直接添加，不重复调度
    ↓
DOM变化3: 修改文本
    ↓
[记录队列] [变化1, 变化2, 变化3]
    ↓
──── [同步代码执行完毕，开始执行微任务] ────
    ↓
📦 打包员开始工作
    ↓
回调收到: [变化1, 变化2, 变化3]  ← 一次性处理所有变化
    ↓
[记录队列] 清空为 []  ← 处理完毕，清空清单
    ↓
（之后如果有新变化，重复以上过程）
```
